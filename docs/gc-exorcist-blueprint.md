## 2. Problem Statement

GC tuning is treated like black magic by most Java developers because:

- GC logs are voluminous, cryptic, and format changes across JVM versions (pre-9 flags vs. unified `-Xlog` since JDK 9)
- Most developers never enable GC logging until there's a production incident — then it's too late
- Interpreting pause times, heap utilization curves, and promotion rates requires deep GC algorithm knowledge
- Existing tools are either SaaS (GCeasy.io), heavyweight GUI (GCViewer, Eclipse MAT), or abandonware
- No tool connects GC behavior back to application code or Spring configuration
- The relationship between GC pauses and thread behavior (see: thread-necromancer) is rarely analyzed together

gc-exorcist fills this gap: a CLI-native, Claude Code-integrated tool that turns GC logs into structured tuning recommendations with concrete JVM flag suggestions.

## 4. Scripts (Shell Layer)

### 4.1 `gc-parser.sh` / `gc-parser.ps1`

**Purpose:** Parse GC logs from any JVM version into a structured format for Claude analysis. This is the heavy lifter — it handles format detection and metric extraction so Claude can focus on interpretation.

**Usage:**

```bash
./scripts/gc-parser.sh <gc-log-file> [--format auto|unified|legacy]
```

**Format detection logic:**

The parser must handle two fundamentally different log formats:

1. **Legacy format (JDK ≤ 8):** Generated by `-XX:+PrintGCDetails -Xloggc:gc.log`
   ```
   2025-03-14T20:45:12.345+0100: 125.456: [GC (Allocation Failure) [PSYoungGen: 524288K->65536K(611840K)] 1048576K->589824K(2010112K), 0.0456789 secs] [Times: user=0.15 sys=0.02, real=0.05 secs]
   ```

2. **Unified format (JDK 9+):** Generated by `-Xlog:gc*:file=gc.log`
   ```
   [2.456s][info][gc           ] GC(3) Pause Young (Normal) (G1 Evacuation Pause) 512M->128M(1024M) 12.345ms
   [2.456s][info][gc,phases    ] GC(3)   Pre Evacuate Collection Set: 0.1ms
   [2.456s][info][gc,phases    ] GC(3)   Merge Heap Roots: 0.2ms
   [2.456s][info][gc,phases    ] GC(3)   Evacuate Collection Set: 10.5ms
   [2.456s][info][gc,phases    ] GC(3)   Post Evacuate Collection Set: 1.2ms
   [2.456s][info][gc,heap      ] GC(3) Eden regions: 25->0(23)
   [2.456s][info][gc,heap      ] GC(3) Survivor regions: 3->4(4)
   [2.456s][info][gc,heap      ] GC(3) Old regions: 12->12
   [2.456s][info][gc,heap      ] GC(3) Humongous regions: 2->1
   ```

**Auto-detection:**
- If lines match `\[\d+\.\d+s\]\[` → unified format
- If lines match `\d{4}-\d{2}-\d{2}T.*\[GC` or `\d+\.\d+:.*\[GC` → legacy format
- If mixed → report error, suggest which portion to analyze

**Parsing implementation notes:**

For the shell parser, use `awk`/`sed` for line-by-line extraction. Key patterns to extract:

- **Unified format pause lines:** `Pause Young|Pause Full|Pause Remark|Pause Cleanup` — extract duration in ms
- **Unified format heap lines:** `gc,heap` tag — extract region counts, sizes
- **Unified format cause:** parenthetical after pause type, e.g., `(G1 Evacuation Pause)`, `(Metadata GC Threshold)`
- **Legacy format pause lines:** `[GC` / `[Full GC` — extract sizes and times from `real=X.XX secs`
- **Allocation rate:** compute from delta of heap-before-GC between consecutive events divided by time delta
- **Promotion rate:** compute from old-gen growth between consecutive Young GC events
- **Percentiles:** sort pause times, compute p50/p95/p99 via array indexing

For complex parsing (especially percentile computation), the script can output raw pause times to a temp file and use `sort -n` + `awk` for percentile calculation.

### 4.2 `gc-capture.sh` / `gc-capture.ps1`

**Purpose:** Enable or adjust GC logging on a running JVM process without restart.

**Commands:**

```bash
# Enable GC logging on a running JVM (JDK 9+)
./scripts/gc-capture.sh enable <PID> [output-dir]
# Runs: jcmd <PID> VM.log output=<file> what=gc*=info decorators=time,uptime,level,tags

# Disable GC logging
./scripts/gc-capture.sh disable <PID>

# Check current GC logging configuration
./scripts/gc-capture.sh status <PID>
# Runs: jcmd <PID> VM.log list

# Get current GC info without log (snapshot)
./scripts/gc-capture.sh snapshot <PID>
# Runs: jcmd <PID> GC.heap_info + jcmd <PID> VM.flags
```

**Notes:**
- Dynamic log configuration only works with JDK 9+ unified logging
- For JDK 8, the script should report that GC logging must be enabled at startup and provide the flags
- Output files go to `.gc-exorcist/logs/` in project root by default

## 5. Skills (Claude Analysis Layer)

### 5.1 `/gc-analyze` — Full GC Log Analysis

```yaml
---
name: gc-analyze
description: Analyze a GC log file and provide tuning recommendations
usage: /gc-analyze <file-path>
arguments:
  - name: file-path
    description: "Path to GC log file (.log, .txt)"
    required: true
---
```

**Skill behavior:**

1. Validate file exists and detect format (unified vs legacy)
2. Run `gc-parser.sh <file-path>` to extract structured metrics
3. Analyze against checklist from reference files:
   - Is the GC algorithm appropriate for this workload?
   - Are pause times acceptable? (< 200ms for latency-sensitive, < 1s for throughput)
   - Is there memory pressure? (frequent Full GCs, high occupancy after GC, upward drift)
   - Are there allocation rate spikes? (bursty allocation patterns)
   - Are there humongous allocation issues? (G1 specific)
   - Is GC overhead acceptable? (< 5% for most apps, < 1% ideal)
   - Are there evacuation failures or to-space exhaustion events?
   - Is the metaspace sized correctly?
   - Safepoint analysis (if data available)
4. Cross-reference with application context:
   - If Spring Boot project detected: check heap size vs typical Spring app needs
   - If Hibernate/JPA present: warn about object creation pressure from lazy loading
   - If batch processing detected: note throughput-oriented GC might be better
5. Output structured report (see Section 7)

### 5.2 `/gc-compare` — Compare Two GC Logs

```yaml
---
name: gc-compare
description: Compare two GC logs to evaluate the impact of a tuning change
usage: /gc-compare <before-log> <after-log>
arguments:
  - name: before-log
    description: "Path to GC log before tuning change"
    required: true
  - name: after-log
    description: "Path to GC log after tuning change"
    required: true
---
```

**Skill behavior:**

1. Parse both logs
2. Compare key metrics side-by-side:
   - Pause time percentiles (p50, p95, p99, max)
   - Full GC frequency and duration
   - GC overhead percentage
   - Heap utilization patterns
   - Allocation/promotion rates
   - Humongous allocation counts
3. Assess: did the tuning change improve things?
4. Highlight regressions (metrics that got worse)
5. Suggest next tuning step based on remaining issues

### 5.3 `/gc-recommend` — JVM Flag Recommendations

```yaml
---
name: gc-recommend
description: Analyze project and generate recommended JVM GC flags
usage: /gc-recommend [gc-log-file]
arguments:
  - name: gc-log-file
    description: "Optional GC log for data-driven recommendations. Without it, recommendations are based on project analysis."
    required: false
---
```

**Skill behavior:**

**Without GC log (project-based):**
1. Analyze `pom.xml`/`build.gradle` for dependencies (Spring Boot, batch, reactive, etc.)
2. Check existing JVM flags in run configs, Dockerfiles, scripts
3. Detect JDK version from build config
4. Recommend GC algorithm + base flags based on:
   - Spring Boot web app → G1GC with moderate pause target
   - Reactive/WebFlux → ZGC or Shenandoah for ultra-low latency
   - Batch processing → Parallel GC for throughput
   - Large heap (> 8GB) → ZGC or G1GC
   - Container/memory-constrained → G1GC with container awareness flags
5. Output a complete JVM flags block ready to copy-paste

**With GC log (data-driven):**
1. Parse log for current behavior
2. Identify the primary issue (latency? throughput? footprint? Full GCs?)
3. Generate targeted flag adjustments with before/after predictions
4. Explain each flag change and expected impact

```yaml
---
name: gc-enable
description: Generate the correct GC logging JVM flags for your JDK version
usage: /gc-enable [--jdk-version <version>] [--detail-level basic|detailed|trace]

  - name: jdk-version

    required: false

```

**Skill behavior:**

1. Detect JDK version from `java -version` or build config
2. Generate appropriate flags:

   **JDK 8 (legacy flags):**
   ```
   -XX:+PrintGCDetails
   -XX:+PrintGCDateStamps
   -XX:+PrintTenuringDistribution
   -XX:+PrintGCApplicationStoppedTime
   -Xloggc:gc.log
   -XX:+UseGCLogFileRotation
   -XX:NumberOfGCLogFiles=5
   -XX:GCLogFileSize=20M
   ```

   **JDK 11/17/21 (unified logging):**
   ```
   # Basic (minimal overhead, good for always-on production)
   -Xlog:gc*:file=gc.log:time,uptime,level,tags:filecount=5,filesize=20M

   # Detailed (standard troubleshooting)
   -Xlog:gc*,gc+ref=debug,gc+phases=debug,gc+age=debug,safepoint:file=gc.log:time,uptime,level,tags:filecount=5,filesize=20M

   # Trace (deep analysis, higher overhead)
   -Xlog:gc*=debug,gc+ergo*=trace,gc+age*=trace,gc+alloc*=trace,safepoint*:file=gc.log:time,uptime,level,tags:filecount=10,filesize=50M
   ```

   **JDK 17+ async logging (recommended):**
   ```
   -Xlog:async
   -Xlog:gc*,safepoint:file=gc.log:time,uptime,level,tags:filecount=5,filesize=20M
   ```

3. Also generate integration snippets:
   - `JAVA_OPTS` for shell scripts
   - Dockerfile `ENTRYPOINT` / `CMD`
   - `spring-boot-maven-plugin` `<jvmArguments>`
   - `application.yml` (for Spring Boot 3.2+ startup arguments)

### 6.2 `references/gc-tuning-rules.md`

Content — decision rules for the analysis:

**Pause Time Rules:**
- p99 > 500ms for web app → CRITICAL: users will notice
- p99 > 200ms for API service → WARNING: may cause timeout cascading
- p99 > 10ms for trading/real-time → consider ZGC/Shenandoah
- Max pause > 5s → CRITICAL: health checks will fail, connections will timeout
- Full GC occurring → WARNING: usually indicates heap pressure or metaspace exhaustion

**Heap Utilization Rules:**
- After-GC occupancy > 70% of max heap → WARNING: insufficient headroom
- After-GC occupancy trending upward over time → CRITICAL: memory leak suspected
- After-GC occupancy < 30% → INFO: heap oversized, consider reducing -Xmx
- Before-GC occupancy consistently at 100% → CRITICAL: heap too small

**GC Overhead Rules:**
- GC overhead > 5% → WARNING: significant time spent in GC
- GC overhead > 10% → CRITICAL: GC thrashing, application barely progressing
- GC overhead > 25% → CRITICAL: near OOM, expect `GC overhead limit exceeded`
- GC overhead < 0.5% → healthy

**Allocation Rate Rules:**
- Sustained > 1 GB/s → WARNING: high allocation pressure, check for object churn
- Bursty spikes > 3× average → WARNING: investigate batch operations or cache warming
- High allocation + high promotion → too many short-lived objects surviving Young GC

**Full GC Rules:**
- Any Full GC in a long-running app → investigate cause
- Full GC cause "Metadata GC Threshold" → increase `-XX:MetaspaceSize` and `-XX:MaxMetaspaceSize`
- Full GC cause "Allocation Failure" → heap too small or fragmentation (CMS/Parallel)
- Full GC cause "System.gc()" → add `-XX:+DisableExplicitGC` or investigate caller (RMI, NIO)
- Full GC cause "G1 Humongous Allocation" → increase region size or reduce allocation size
- Full GC cause "Evacuation Failure" / "To-space exhausted" → heap too small, increase -Xmx or tune G1

**G1-Specific Rules:**
- Humongous allocations > 10/min → WARNING: objects > 50% of region size (default 1-32MB)
- Fix: increase region size `-XX:G1HeapRegionSize=16m` or reduce allocation size
- Mixed GC not reclaiming enough → adjust `-XX:G1MixedGCLiveThresholdPercent`
- Evacuation failures → increase `-XX:G1ReservePercent` (default 10%)

**Container/Cloud Rules:**
- Always set `-XX:MaxRAMPercentage=75.0` (not -Xmx in absolute terms) in containers
- Always verify `-XX:+UseContainerSupport` is active (default since JDK 10)
- Check that `-XX:ActiveProcessorCount` is not artificially low

### 6.3 `references/gc-log-formats.md`

Content — format reference for parsing:

**Unified log line structure (JDK 9+):**
```
[<uptime>][<level>][<tags>] <message>
```
Example: `[2.456s][info][gc] GC(3) Pause Young (Normal) (G1 Evacuation Pause) 512M->128M(1024M) 12.345ms`

Key tag combinations:
- `gc` — main GC events (pauses, concurrent cycles)
- `gc,start` — GC event start markers
- `gc,phases` — sub-phase timings within a GC event
- `gc,heap` — heap region details (G1: Eden, Survivor, Old, Humongous)
- `gc,metaspace` — metaspace usage
- `gc,age` — tenuring distribution
- `gc,alloc` — allocation-related events
- `gc,ergo` — ergonomic (adaptive) decisions
- `gc,humongous` — humongous allocation events
- `gc,ref` — reference processing (soft, weak, phantom, JNI)
- `safepoint` — safepoint timing (not a gc tag but critical for analysis)

**Pause type patterns to match:**
```
Pause Young (Normal)
Pause Young (Concurrent Start)
Pause Young (Prepare Mixed)
Pause Young (Mixed)
Pause Full (Allocation Failure)
Pause Full (Metadata GC Threshold)
Pause Full (System.gc())
Pause Full (G1 Humongous Allocation)
Pause Full (G1 Compaction Pause)
Pause Remark
Pause Cleanup
Concurrent Mark Cycle (<duration>)
Concurrent Undo Cycle (<duration>)
```

**Legacy format (JDK ≤ 8) patterns:**
```
[GC (Allocation Failure) [PSYoungGen: <before>K-><after>K(<capacity>K)] <heap-before>K-><heap-after>K(<heap-capacity>K), <pause> secs]
[Full GC (Metadata GC Threshold) [PSYoungGen: ...] [ParOldGen: ...] <heap-before>K-><heap-after>K(<heap-capacity>K), [Metaspace: ...], <pause> secs]
```

**Legacy flag → Unified logging mapping table:**

| Legacy Flag (JDK ≤ 8)                 | Unified Equivalent (JDK 9+)           |
|---------------------------------------|---------------------------------------|
| `-XX:+PrintGC`                        | `-Xlog:gc`                            |
| `-XX:+PrintGCDetails`                 | `-Xlog:gc*`                           |
| `-XX:+PrintGCDateStamps`             | decorator `:time`                     |
| `-XX:+PrintGCTimeStamps`            | decorator `:uptime`                   |
| `-XX:+PrintTenuringDistribution`     | `-Xlog:gc+age*=debug`                |
| `-XX:+PrintReferenceGC`             | `-Xlog:gc+ref=debug`                 |
| `-XX:+PrintGCApplicationStoppedTime` | `-Xlog:safepoint`                    |
| `-XX:+PrintAdaptiveSizePolicy`       | `-Xlog:gc+ergo*=trace`              |
| `-Xloggc:<file>`                     | `-Xlog:gc*:file=<file>`             |

### 6.4 `references/spring-gc-patterns.md`

Content — Spring Boot specific GC patterns:

**Spring Boot Startup GC Pattern:**
- During startup: high allocation rate (classpath scanning, bean creation, auto-configuration)
- Typically 5-20 Young GCs during startup, possibly 1-2 Full GCs if metaspace is undersized
- After startup: allocation rate should drop significantly
- If allocation rate remains high post-startup → investigate scheduled tasks, cache warming, open-in-view

**Common Spring Boot GC Pitfalls:**

1. **Hibernate Second-Level Cache flooding Old Gen:**
   Objects promoted from cache → long-lived → fill Old Gen → frequent Mixed/Full GCs.
   Signs: old gen occupancy steadily climbing, high promotion rate.
   Fix: Configure cache max size, use off-heap cache (EhCache/Caffeine), set eviction policy.

2. **Jackson serialization object churn:**
   Each request creates temporary objects for JSON serialization.
   Signs: very high Young GC frequency, high allocation rate.
   Fix: Reuse ObjectMapper, consider streaming serialization, use DTOs instead of entity proxies.

3. **Spring Boot DevTools class reloading:**
   In dev mode, class reloading creates metaspace pressure.
   Signs: `Metadata GC Threshold` Full GCs during development.
   Fix: Increase `-XX:MetaspaceSize=256m -XX:MaxMetaspaceSize=512m` for dev profile.

4. **Actuator /heapdump endpoint:**
   Triggering a heap dump creates massive allocation spike.
   Signs: sudden Full GC correlated with actuator endpoint access.
   Fix: Restrict access to heapdump endpoint, use async dumping if available.

5. **Spring Batch large dataset processing:**
   Processing millions of records without chunk-aware GC tuning.
   Signs: sawtooth heap pattern with increasing baseline (leak-like).
   Fix: Use Parallel GC for batch jobs, configure `-XX:+UseParallelGC` for batch profiles, ensure chunk processing clears references properly.

6. **WebFlux backpressure failure:**
   Reactive streams without proper backpressure → unbounded buffering.
   Signs: heap grows unboundedly, frequent Full GCs, eventually OOM.
   Fix: Configure buffer limits, use `limitRate()`, check `spring.codec.max-in-memory-size`.

**Container-Specific Spring Boot GC Recommendations:**

### 6.5 `references/jvm-flag-catalog.md`

Content — complete catalog of GC-related JVM flags with defaults and recommendations:

**Heap Sizing:**
- `-Xms` / `-Xmx` — initial/max heap. Recommendation: set equal for production (avoids resize pauses).
- `-XX:MaxRAMPercentage=75.0` — for containers. Leaves 25% for JVM overhead, thread stacks, native memory.
- `-XX:MinRAMPercentage` — only applies when max heap < ~200MB.
- `-XX:NewRatio` — Old:Young ratio. Default 2 (1/3 young, 2/3 old). Rarely needs tuning with G1.
- `-XX:MetaspaceSize=256m` — initial metaspace (triggers Full GC when exceeded). Set high for Spring Boot.
- `-XX:MaxMetaspaceSize=512m` — hard limit. Spring Boot apps easily use 150-300M.

**G1GC Tuning:**
- `-XX:MaxGCPauseMillis=200` — pause time target (default 200ms). Lower = more frequent, shorter GCs.
- `-XX:G1HeapRegionSize=<n>m` — region size (1-32MB, auto-selected). Increase if humongous allocations.
- `-XX:G1ReservePercent=10` — reserved heap for evacuation. Increase if evacuation failures.
- `-XX:G1MixedGCLiveThresholdPercent=85` — region reclamation threshold. Lower = more aggressive.
- `-XX:InitiatingHeapOccupancyPercent=45` — when to start concurrent marking. Lower = earlier, safer.
- `-XX:G1NewSizePercent=5` / `-XX:G1MaxNewSizePercent=60` — young gen bounds.

**ZGC Tuning (JDK 17+):**
- `-XX:+UseZGC` — enable ZGC.
- `-XX:+ZGenerational` — generational ZGC (JDK 21+, default in JDK 23). Recommended.
- `-XX:SoftMaxHeapSize` — target heap size (ZGC will try to stay below, can exceed if needed).
- ZGC needs minimal tuning — set `-Xmx` and let it self-tune.

**Shenandoah Tuning:**
- `-XX:+UseShenandoahGC` — enable Shenandoah (OpenJDK/Corretto/Zulu only, not Oracle JDK).
- `-XX:ShenandoahGCHeuristics=adaptive` — default heuristic.
- Minimal tuning needed — primarily tune `-Xmx`.

**Diagnostic Flags:**
- `-XX:+HeapDumpOnOutOfMemoryError` — always enable in production.
- `-XX:HeapDumpPath=/path/` — where to write heap dumps.
- `-XX:+DisableExplicitGC` — prevent `System.gc()` calls (e.g., from RMI, NIO).
- `-XX:+ExitOnOutOfMemoryError` — crash JVM on OOM (good for containers, enables restart).
- `-XX:NativeMemoryTracking=summary` — track native memory (small overhead).

### 6.6 `references/gc-selection-guide.md`

Content — decision tree for GC algorithm selection:

```
START
  │
  ├── Heap < 256MB? → Serial GC
  │
  ├── Latency critical (< 10ms p99)? → ZGC (JDK 17+) or Shenandoah
  │     └── Oracle JDK? → ZGC (Shenandoah not available)
  │     └── OpenJDK/Corretto? → ZGC or Shenandoah (both fine)
  │
  ├── Throughput critical (batch, offline)? → Parallel GC
  │
  ├── General purpose web/API? → G1GC (default since JDK 9)
  │     └── Heap > 8GB? → Consider ZGC
  │     └── Frequent humongous allocations? → Increase G1HeapRegionSize
  │
  └── Container with < 1GB memory? → Serial GC or G1GC
        └── G1GC with reduced region count may underperform → test both
```

## 11. Benchmark Plan

### Fixture Setup

Generate GC logs from a Spring Boot 3.2 application with intentionally problematic configurations:

1. **Undersized heap** — `-Xmx256m` for a 512M working set → frequent Full GCs
2. **Humongous allocations** — allocate 4MB+ byte arrays in a loop → G1 humongous events
3. **Metaspace exhaustion** — dynamic class generation (Groovy/CGLIB) → Metadata GC Threshold Full GCs
4. **Memory leak simulation** — growing list never cleared → upward occupancy drift
5. **High allocation rate** — JSON serialization of large objects in tight loop → young GC pressure
6. **Evacuation failure** — near-full heap with fragmented old gen → to-space exhaustion
7. **System.gc() abuse** — explicit GC calls from code → unnecessary Full GCs

Capture GC logs for each scenario (15-30 minutes of runtime), then test skills against them.

## 13. Synergies

### thread-necromancer
- Thread dumps showing "all threads paused" → gc-exorcist confirms GC pause at that timestamp
- Connection pool exhaustion visible in threads → gc-exorcist shows if GC pause caused the initial backlog
- Combined workflow: `/thread-watch` captures temporal data → correlate with `/gc-analyze` timeline

### heap-seance
- Heap analysis shows what objects consume memory → gc-exorcist shows how GC handles that memory
- Memory leak detected by heap-seance → gc-exorcist confirms upward occupancy drift in GC logs
- Combined: "your Old Gen is full of Hibernate proxies" (heap-seance) + "that's causing evacuation failures" (gc-exorcist)

### spring-grimoire
- `/spring-jpa-audit` finds lazy loading issues → gc-exorcist shows the allocation rate impact
- `/spring-config-audit` (planned) checks pool/timeout configs → gc-exorcist validates GC impact of those settings
- `/spring-dockerfile` generates Dockerfile → gc-exorcist's hook validates that GC flags are included
